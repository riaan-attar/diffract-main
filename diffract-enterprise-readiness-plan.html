Diffract — Enterprise Readiness & Hardening roadmap

Status: Draft / plan only (no code changes)  •  Created: 2026-06-03  •  Current maturity: ~2/5 — solid agent core, prototype/demo shell  •  Companion: diffract-multitenant-payments-plan.html

Where we are The agent runtime (OpenShell / NemoClaw / Hermes) is strong on isolation — egress allowlisting, secrets-not-in-box, Landlock FS, config integrity. The Diffract productization layer (diffractui + setup.sh + ops) is demo-grade: no auth, the API key is passed in a URL query param, services shell out with sudo / shell:true, no tests / CI / monitoring, single-VPS, manual prod edits. This session, the headline feature (web-UI agent chat) was fully broken and fixed only by hand-patching prod — a fair barometer of current operational maturity.
Approach Two interleaved tracks: (A) Harden & productionize what exists, and (B) build the SaaS / multi-tenant / payments layer (detailed in the companion architecture doc). Sequenced into 5 phases, each gated by the milestone it unlocks — so we never expose a paid/public surface before the controls that protect it exist, and never build the 500-tenant version before client #2.

Guiding principles

  • Security before exposure — no public or paid URL until auth + secret hygiene exist.
  • Automate before scale — no more hand-edits on prod; provisioning must be reproducible and health-checked.
  • Defense in depth — keep OpenShell's agent governance; add layers, don't replace them.
  • Right-size to client count — build the minimal safe version for the next client, not for 500.
  • Make outages page you, not the customer — observability is a feature.

Phased roadmap

0 Stop the bleeding blocking for ANY external user · ~days

These are non-negotiable before anyone outside your team touches a URL — paid or not.

  • Auth on diffractui — gate every page + all mutating APIs (/api/deploy, destroy, /api/forward). Minimal now: one admin login / signed session. Per-tenant auth comes in Phase 2.
  • Get the API key out of the URL — it's currently searchParams.get("apiKey") on a GET (lands in access logs, proxies, history). Move to a POST body, pass to onboard via env/stdin only, never log it.
  • De-privilege the shell-outs — replace shell:true string interpolation and sudo systemctl restart with a constrained helper (fixed command allowlist / polkit rule) and validate inputs (sandbox name, provider) against a strict pattern. Today this is a command-injection + privilege-escalation surface.
  • Uptime + health alerting — a check that asserts dashboard:9119 (chat WS = 101), agent API:8642 /health, and diffractui:3000, and pages you on failure. (This is exactly the outage we hit this session.)
  • Fix latent bugs — gateway-token reads the wrong /sandbox/.openclaw/… path on a Hermes box; audit similar OpenClaw/Hermes mismatches.
Exit: no unauthenticated control surface; no secret in a URL; no unconstrained sudo; you get paged before a customer notices an outage.

1 Pilot-ready · single tenant, design partners · ~weeks

  • CI/CD for diffractui — lint + typecheck + a smoke test; deploy via pipeline, not by editing files on the box.
  • Reproducible, idempotent provisioning — replace the fragile setup.sh + systemd + socat dance with a health-checked install where the dashboard-with---tui launch (this session's fix) is baked in, plus a post-deploy smoke test that asserts a real agent round-trip (WS 101 + a "PONG").
  • Secrets at rest — provider keys/tokens into a secret store; correct file perms; restricted access. (Vault/KMS can wait, but stop leaving keys loose.)
  • Structured logging + error tracking — Sentry/OTel; centralized logs for diffractui, gateway, dashboard.
  • Backups + documented restore of config/state.
Exit: a new environment can be stood up and verified automatically; failures are observable; a friendly customer can self-serve safely.

2 Multi-tenant foundation see architecture doc · ~weeks–months

  • Tenant DB as source of truth (not sandboxes.json) — subdomain, status, backend, owner.
  • Server-enforced per-tenant authorization — every API call verifies the caller owns the sandbox it touches. (On shared infra this is load-bearing security, not polish.)
  • Tenant-aware diffractui — resolve sandbox from the subdomain; emit the agent under the chosen path prefix.
  • Central ingress — wildcard *.diffraction.in + DNS-01 wildcard TLS; retire the hardcoded *.hstgr.cloud host.
  • RBAC roles (Viewer / Operator / Admin / Security / Billing) + SSO / OIDC (Okta, Azure AD, Google) — enterprise table stakes.
Exit: multiple isolated tenants on tenant.diffraction.in, each access-controlled; SSO works.

3 Monetization + automated provisioning see architecture doc · ~months

  • Dodo Payments — checkout → signed, idempotent webhook → provisioner → tenant registry → routing; deprovision on cancel/refund.
  • Central LiteLLM inference plane — since Diffract owns the key: per-tenant virtual keys with fail-closed budgets, rate limits, and model allowlists; real upstream key kept off the client hosts. (The spend cap is live from client #1.)
  • Usage metering → billing reconciliation + quota enforcement + anomaly alerts.
Exit: a client can pay and get a working, cost-capped environment with no manual steps; cancellation tears it down.

4 Enterprise GA — compliance, resilience, scale · ~months

  • Audit logging — immutable, per-tenant action trail + audit UI (procurement requirement).
  • Compliance — SOC 2 Type II posture, third-party pentest, data-residency story; document the OpenShell control mapping (egress / FS / secrets / config).
  • HA / no SPOF — redundant control plane + ingress; multi-host sandbox pool with placement/scheduling; warm pool for fast provisioning.
  • DR + operations — RTO/RPO targets, restore drills, on-call, status page, SLAs.
  • Dependency-maturity gate — NemoClaw is alpha ("not production-ready" per NVIDIA). Track its path to stable; pin versions; have a fork/patch strategy and an upstream-risk plan.
  • Legal / branding — review the rebrand of NVIDIA (Apache-2.0) + Nous (MIT) forks: attribution, trademark, and the "OpenClaw / OpenShell / NemoClaw" renames before commercial sale.
Exit: survives an enterprise security review, an availability commitment, and a procurement/legal review.

Definition of "enterprise-ready" (the GA gate)

  • SSO/OIDC + RBAC; every mutating action authenticated & authorized per tenant
  • No secret ever in a URL/log; secrets in a managed store; key never on client compute
  • Tenant isolation validated (data + compute + credentials), backed by OpenShell governance
  • Immutable per-tenant audit log + exportable trail
  • Automated, idempotent provisioning + deprovisioning with post-deploy verification
  • Fail-closed per-tenant spend caps + rate limits (Diffract-owned key)
  • Monitoring/alerting/on-call; documented SLA, status page, DR with tested restore
  • No single point of failure for control plane + ingress
  • CI gates on every change; zero manual prod edits
  • SOC 2 posture + pentest; licensing/trademark cleared
  • Dependency-maturity risk owned (NemoClaw alpha → pinned/forked/tracked)
Honest timeline Phase 0 is days; Phase 1 is weeks; Phases 2–4 are a multi-month, multi-workstream effort — and "enterprise-stable" is partly gated on upstream you don't fully control (NemoClaw alpha). Treat P0/P1 as the immediate, fully-in-your-control work that de-risks everything else.

Diffract — Multi-tenant + Dodo Payments architecture plan

Status: Draft / design only (no code changes)  •  Created: 2026-06-03  •  Scope: per-client subdomains on diffraction.in, Dodo Payments, automated provisioning

Context Builds on the current single-tenant stack: one VPS, Caddy fronting diffractui (:3000), the Hermes dashboard (:9119) and agent API (:8642), with the model API key held in OpenShell's provider store on the host and injected at the egress proxy. Goal: turn this into a paid, multi-tenant product where each client gets client.diffraction.in/ (Diffract dashboard) and client.diffraction.in/client/agent-chat (their agent), provisioned automatically on payment.

0. Two decisions to make first (everything follows)

Decision A — Isolation model load-bearing

VPS-per-client (recommended v1)Container-per-client (shared VPS)
IsolationFull OS/host isolation per clientOpenShell isolates agents, but host (Caddy, docker socket, OpenShell key store) is shared
Reuse of current stacksetup.sh/Caddy/diffractui run unchanged, parameterized by domainRequires tenant-aware diffractui, port multiplexing, host hardening
Cost / densityHigher $/clientCheaper, denser
Provisioning timeMinutes (need a "preparing…" state or warm pool)Seconds
Recommendation VPS-per-client for v1. Two facts push hard that way: (1) the product is security isolation — co-tenanting agents on one host undercuts the pitch; (2) the model API keys live in OpenShell's provider store on the host and are injected at the egress proxy — on a shared host, every client's credentials sit in one store. Container-per-client is the right density optimization later, after auth + host hardening exist.

Decision B — Who pays for inference?

  • (a) Client brings their own API key (recommended) — onboard already collects it. Dodo just gates the Diffract service/seat. Simple.
  • (b) Diffract provides metered inference on its own key — needs usage metering, quotas, overage billing, abuse protection. Much larger build.
Recommendation (a) BYO-key — matches "onboard Diffract on his business" and keeps Dodo to subscription gating. The rest of this plan assumes VPS-per-client + BYO-key; choosing differently changes §4–§6 materially.

The central Caddy ingress is the key simplifier: it terminates *.diffraction.in, looks up the subdomain in the tenant DB, and reverse-proxies to that client's VPS. Result: one wildcard DNS record + one wildcard cert — no per-client DNS automation.

2. Domain, DNS, TLS (replacing the hardcoded host)

Verify first Do we have programmatic DNS control for diffraction.in (a provider API token)? This gates the whole automation story.
  • DNS: one wildcard record *.diffraction.in → ingress IP. New clients need zero DNS changes — the subdomain only has to exist in the tenant DB.
  • TLS: wildcard cert for *.diffraction.in via DNS-01 on the ingress. Prefer this over Caddy's on-demand-per-subdomain TLS, which hits Let's Encrypt rate limits at scale and needs an ask endpoint to bound issuance.
  • Replace srv1670849.hstgr.cloud: in the target model the per-client VPS no longer needs a public Caddy (the ingress does TLS + routing). Each VPS just exposes 3000 / 9119 / 8642 to the ingress over a private network / firewall allowlisted to the ingress IP. setup.sh becomes "install + bind services," not "own a public domain."
  • Immediate step for the existing client: point e.g. client.diffraction.in at the current VPS now (directly or via the ingress) to retire the *.hstgr.cloud URL without waiting for the full build.

3. Per-client routing config, not new architecture

A generalization of the / + /agent split already fixed and verified. At the ingress, per Host: <tenant>.diffraction.in:

Path→ backendNotes

/client/agent-chat/*client VPS :9119 (Hermes dashboard)strip prefix; set X-Forwarded-Prefix: /client/agent-chat

The Hermes dashboard already honors any prefix (X-Forwarded-Prefix → window.__HERMES_BASE_PATH__), and its chat WS becomes /client/agent-chat/api/ws — the exact mechanism exercised this session. Mounting the agent at /client/agent-chat is a one-line prefix change, not new work.

Clarify (non-blocking) In client.diffraction.in/client/agent-chat, is /client/ a literal brand segment, or the tenant name (acme.diffraction.in/acme/agent-chat)? Either way it's one prefix-config line.

4. Tenant registry / DB (source of truth — not sandboxes.json)

A small central database the control plane and ingress both read. Minimum fields:

  • subdomain (e.g. acme), display_name
  • status: pending_payment → provisioning → active → suspended → deprovisioned
  • backend — VPS id / IP / port set (3000/9119/8642)
  • dodo_customer_id, dodo_subscription_id, plan, current_period_end
  • provisioning_state + last error (for rollback/retry)
  • auth linkage (tenant owner + members), timestamps

~/.nemoclaw/sandboxes.json stays a per-VPS local detail; it is not the multi-tenant source of truth.

5. Provisioning pipeline the actual core

Dodo webhook (payment / subscription active)
  → verify signature  → idempotency check (dedupe retries)
  → create tenant row (status=provisioning)
  → provision backend:  VPS API create (or claim from warm pool)
                        cloud-init runs the Diffract installer (BYO-key from onboard)
  → register subdomain → backend in tenant DB   (ingress picks it up; no DNS call)
  → health-check (diffractui:3000, dashboard:9119 chat WS 101, /v1/health)
  → status=active  → email client their URL
  ── on any failure: rollback + alert; client sees "preparing/failed", not a broken page
On cancel/refund webhook → status=suspended → (grace) → deprovision VPS, free subdomain

Must-haves, each a real task: webhook signature verification, idempotency (Dodo retries/duplicates), rollback + alerting when payment succeeds but provisioning fails, and deprovision on cancellation. Provisioning a fresh VPS programmatically needs a provider with a real API (Hetzner/DO/Lightsail are friendlier than Hostinger here) plus a pre-baked image/snapshot of the Diffract stack (or cloud-init running the installer); a warm pool of 1–2 ready VPSs hides the multi-minute spin-up.

6. Dodo Payments integration

Standard Merchant-of-Record flow:

  1. Checkout — a Dodo product/subscription + hosted payment link (or embedded checkout) on the signup page; collect desired subdomain at/after checkout.
  2. Webhooks — on payment/subscription active → trigger provisioning (§5); on cancelled/refunded → suspend/deprovision.
  3. Lifecycle — store subscription_id; reconcile renewals; suspend on dunning failure.
Verify, don't reconstruct Pull the exact event names and the webhook signature scheme from Dodo's current docs at build time — those are the details that silently break. Dodo is an MoR, so it also handles tax/VAT (a plus for a global SaaS).

7. Auth — a hard prerequisite for go-live currently missing

Blocking Today diffractui has no authentication: /api/deploy, /api/status?…action=destroy, and /api/forward are open and shell out with sudo. The moment client.diffraction.in is public and paid, anyone with the URL controls that tenant's sandbox. Per-tenant login is blocking, not polish (mulch notes already list SSO/RBAC as planned-not-built).

Minimum v1: a tenant-scoped login gating the dashboard + all mutating APIs. The agent dashboard already has its own session token; the Diffract control surface does not.

8. diffractui changes

  • VPS-per-client keeps diffractui single-tenant per box → minimal change (it already resolves the one defaultSandbox). Additions: auth (above) + reading its public URL/prefix from env so it emits /client/agent-chat instead of hardcoding /agent.
  • Only if container-per-client is chosen later: diffractui must become tenant-aware — resolve the sandbox from the Host header against the registry on every call. Larger change.

9. Phased rollout (suggested order)

  1. Domain cutover — wildcard *.diffraction.in + ingress + wildcard TLS; move the existing client off *.hstgr.cloud. (Low risk, immediate value.)
  2. Tenant DB + ingress lookup — subdomain → backend.
  3. Auth on diffractui — gate before anything is public/paid.
  4. Provisioning automation — VPS API + cloud-init + health checks + rollback (manual trigger first).
  5. Dodo integration — checkout + webhooks wired to the provisioner; deprovision on cancel.
  6. Polish — warm pool, billing portal, suspension/grace, observability.

10. Decisions needed
